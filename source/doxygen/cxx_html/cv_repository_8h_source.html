44 // Class cvRepository provides a mechanism for storing objects and
45 // managing access to those objects. By using cvRepository, other
46 // modules (e.g. in a Tcl environment) can depend on this shared
47 // resource instead of depending directly on each other. Modules may
48 // read or write to the repository without needing to know about which
49 // other modules create or use those objects. This improves
50 // abstraction among those modules, and potentially enables modules to
51 // be included or excluded from specific builds of, say, a Tcl
52 // application without disabling other modules.
53 
54 // Using cvRepository also enables applications to manage data in-core
55 // instead of using other mechanisms (e.g. files... yeech...) for
56 // communicating between modules. Modules share data in-core,
57 // eliminating all of the following:
58 // - redundant in-core representations
59 // - pollution of the filespace
60 // - unnecessary file I/O
61 // - unnecessary data structure allocation and initialization
62 
63 // Clients should allocate objects to be put in the repository, but
64 // should NEVER delete those objects. cvRepository assumes memory
65 // management responsibility for any object once it is registered.
66 
67 // As far as the object name iterator is concerned, we will follow the
68 // rules set by the Tcl hash table package, which is what we're using
69 // to implement this table. Those rules state that it is not safe to
70 // make changes to a table when a search (i.e. iterator) is active.
71 // Therefore, after any Register or UnRegister operation, InitIterator
72 // will need to be called anew to traverse the object names in the
73 // repository. If a query is made (i.e. GetNextName) when that
74 // condition is not satisfied (i.e. the last call to InitIterator was
75 // prior to either a more recent Register or UnRegister), then
76 // GetNextName returns NULL.
